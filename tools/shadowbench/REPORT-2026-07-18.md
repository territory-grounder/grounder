## (d) Qualitative findings (worked example: IFRNLLEI01PRD-1824)

**Incident.** `dc1librespeed01` — Device Down (SNMP unreachable). Subject is
LXC `101020206` on node `dc1pve01`.

**What the predecessor did well.** It did *not* reflexively restart. It
independently confirmed identity/status via NetBox + `pve_guest_status`, then read
the **PVE task log** and found a clean `vzshutdown` by `root@pam` at
2026-07-18T14:24:18Z (status OK) — distinguishing a **deliberate operator
shutdown** from a crash/OOM/host-reboot. Corroborating signals: host stayed up,
`onboot=1` would have auto-restarted on a host reboot, `ha.managed=0` (no cluster
auto-restart), and a sibling helper container (`dc1slurpit01`) was shut down
40s earlier — a coordinated maintenance action. It correctly concluded **no
remediation warranted** and landed on **POLL_PAUSE** (risk low, auto_approved=0,
auto_proceed_on_timeout=0, sms_required=0). Judge 4.75/5; only ding: it did not
surface explicit poll options (offer restart-vs-leave-down) before concluding.

### Finding 1 — The predecessor's shadow gate **degraded its own investigation**

The default-deny shadow hook **blocked 3 raw-shell investigation commands** during
this triage:

- 2× NetBox `curl -H 'Authorization: Token …' | jq/python` → rejected by the
  egress guard (2 `tool_guardrail_rejection` rows in `event_log`).
- 1× `git -C …/production log … ; grep -rl librespeed …/memory/` → blocked by
  `mutation-shadow-gate` `blocking_segment`.

The agent **recovered** by falling back to MCP tools (NetBox MCP +
`pve_node_tasks`) and still reached a fully grounded, evidence-backed conclusion.
So for *this* incident the gate degraded the *path* but did not prevent grounding.
**The risk is structural, not hypothetical:** an investigation that had *no* MCP
fallback for a blocked shell probe would have been forced to conclude on thinner
evidence. TG's read-only investigation tools are native (no shell egress to be
denied), which avoids this failure mode by construction — but this is an
**architectural argument, not yet an evidenced win**, because TG did not triage
this incident.

> On the strict reading of the benchmark rule (a NetBox curl / git log / grep that
> is default-denied ⇒ `couldGround=false`), this incident is recorded
> `couldGround=false`. The nuance above is why that strict flag **understates**
> the actual outcome: grounding *did* succeed via fallback.

### Finding 2 — Credential-in-log hygiene

The shadow log line for this session **embeds a LIVE NetBox token**
(`Authorization: Token <hex>`). The extractor `redact()`s it to `<REDACTED>` in
every emitted field; verified 0 occurrences of the token in JSON output, and the
raw `full_command` that carries the token is **never surfaced** in any record.
**Finding for the estate:** secrets are landing in a plaintext shadow log on the
gateway runner box (`dc1claude01`). That is a real exposure independent of this
benchmark and should be raised with the owner. The benchmark harness is safe; the
*source log* is not.

### Finding 3 — TG's nearest-miss row (context, excluded by filter)

The only TG `session_triage` row after 17:00 UTC is
`am-ChaosFindingsHarvestStale-dc1claude01` (17:21:43). It is **Alertmanager-**
**sourced** (`am-` prefix), not LibreNMS, so the benchmark's `%librenms%` filter
excludes it. Its conclusion is nonetheless *correct*: it declines action because
`dc1claude01` is unobservable in LibreNMS/syslog (`outcome=no-proposal:stop`,
evidenceCount=5, judge mean 4.25). This shows TG *is* producing sane grounded
triage in the window — just not on a LibreNMS incident that pairs with the
predecessor.

## (e) Honest caveats — what is and is NOT proven

**NOT proven:**

- **No head-to-head result exists.** 0 aligned pairs ⇒ no dimension winner, no
  aggregate. Any claim that "TG beats/loses to the predecessor" is unsupported by
  this window.
- TG's architectural advantages (native read-only tools; incident-driven S/N) are
  **design arguments**, corroborated by *this incident's* shape but **not** by a
  paired measurement.

**Proven / supported by data:**

- The predecessor did exactly 1 real agentic triage in the window, and did it well
  (4.75/5).
- The predecessor's window is 97.3% scheduled-cron noise.
- The predecessor's shadow gate blocked 3 of its own investigation commands on the
  one real incident.
- A live credential is present in the predecessor's shadow log.
- TG produced 0 LibreNMS triage rows in the shadow window; its 1 in-window row is
  Alertmanager-sourced and correctly declines action.

**Caveats on method:**

1. **Sparse data.** Shadow mode started *today* (17:20 UTC). The comparable window
   is ~82 minutes. One incident is not a benchmark; it is a first data point.
2. **Scope alignment.** Both systems observe the shared `dc1` estate, so the
   *estate* is aligned — but the *incident sets* did not overlap in this window.
   The out-of-window TG replay corpus (~26 `librenms-dc1-replay-*` rows,
   02:20–02:22 UTC) is **deliberately not counted** here; to include it, rerun
   with `SHADOW_FROM` set earlier and re-pair.
3. **Blind-judge fairness.** The two systems' native judges use **different
   rubrics** (see the rubric-mismatch note under the table). A single blind judge
   must re-score both trajectories on one shared rubric before any paired number is
   credible. Native scores are reported for context only, never head-to-head.
4. **Do not cherry-pick.** The predecessor's single incident is genuinely good;
   this report does **not** discount it. TG's absence on that incident is a
   **coverage gap to close**, reported as such — not spun as a TG win. The correct
   read of 2026-07-18 is: *insufficient overlap to compare; both systems behaved
   correctly on the incidents they did see.*
