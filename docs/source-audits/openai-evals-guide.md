# OpenAI — Evals Guide

> **[O] audit-overlay · TG-38 source benchmark.** Category: **OpenAI** · TG adherence: **3.4 / 5** · Practices audited: **11** (implemented 3 · partial 7 · gap 1 · N/A-by-design 0).

## Verdict

TG implements the eval-driven ethos deeply and in places EXCEEDS the source: a durable Temporal judge cron scoring every production session, an offline-gated → Welch-t A/B → post-graduation regression-watch skill flywheel (the predecessor's dark/starved pipeline rebuilt with its failure modes designed out), smooth 1..5 scoring throughout, and a rigorous statistical negative control (degree-preserving shuffled null, INV-22) that is stronger than the guide's should-not-trigger notion. Where it falls short is operational breadth: the LLM-judge corpus eval only runs manually on-box (no CI eval gate on prompt/skill changes), the sealed holdout and its 20-pt overfitting deploy-gate are sealed-by-construction but Phase-4/planned rather than running, there are no judge calibration splits (only a frontier cross-check anchor), the grader taxonomy lacks string/similarity/weighted-multi graders, and the corpus has zero negative-control (should-not-act) incidents. Net adherence ~3.4/5: excellent on the grounding-specific mechanisms TG was built around, thin on the day-two eval operations the guide emphasizes.

## Top gaps

- Wire a CI eval gate that runs the LLM-judge corpus (or a fixed mechanical grounding score) on prompt/skill/agent changes — today it only runs manually on-box
- Make the sealed holdout operational: an actual holdout corpus + monthly runner + the >20pt regression-vs-holdout overfitting deploy gate (currently protected-by-construction but never run)
- Add negative-control incidents to the eval corpus (should-NOT-act / stale / admin-disabled / out-of-scope) to measure false-positive triage and skill invocation
- Add LLM-judge calibration: a labeled train/validation/test split with TPR/TNR floors and/or few-shot exemplars in the judge prompt
