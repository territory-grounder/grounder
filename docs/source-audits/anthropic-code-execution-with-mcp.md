# Anthropic — Code Execution with MCP

> **[O] audit-overlay · TG-38 source benchmark.** Category: **Anthropic** · TG adherence: **1.5 / 5** · Practices audited: **7** (implemented 1 · partial 3 · gap 2 · N/A-by-design 1).

## Verdict

TG is architecturally the anti-pattern this source contrasts against: a native Go ReAct loop that emits one typed JSON directive per model turn and dispatches a single read-only tool via an exact allowlist lookup — many sequential round-trips, no code orchestration, no batching. The headline mechanism (the agent writes code that runs in a sandbox to orchestrate MCP tools) is legitimately N/A-by-design: INV-08 makes it a foundational invariant that no model token ever becomes control flow/command/query, so executing model-authored code is categorically excluded (and the Completer never spawns a subprocess). Where TG does well is the one benefit code-execution is meant to deliver without needing code-execution: server-side data filtering — tools return compact, secret-stripped summaries so raw payloads never reach the model. The two applicable headline practices (reduce round-trips; scale MCP tool surfaces) are weak: the loop is strictly sequential, all tool names are flat-listed in every system prompt (currently masked by a 4-tool set), observations accumulate in context with no compaction, and the agent's own completion path doesn't even parse the token-usage block, so any "token savings" is unmeasured.

## Top gaps

- Concurrent/batched read-only tool dispatch per directive (errgroup) to collapse sequential round-trips
- On-demand / progressive tool-definition loading (namespaced catalog + list/describe step) so the tool surface scales past a handful
- Observation compaction/eviction — prune or reference-by-ID superseded results so context doesn't grow linearly with cycles
- Persisted per-session token+cost ledger on the agent's gateway path (adapters/model.Gateway ignores the usage block today) to make any savings measurable
