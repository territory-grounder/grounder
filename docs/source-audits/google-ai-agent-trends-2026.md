# Google — AI Agent Trends 2026 (atomic transactions, agent undo stacks / reversibility, enterprise adoption)

> **[O] audit-overlay · TG-38 source benchmark.** Category: **Google** · TG adherence: **3.5 / 5** · Practices audited: **7** (implemented 4 · partial 2 · gap 1 · N/A-by-design 0).

## Verdict

TG is strong on most ingredients the Google source calls for and weak on the one it names first. Reversibility-as-a-gating-property (irreversible/destructive/stateful ops clamp to POLL_PAUSE with a server-side override), idempotent tools/activities, durable Temporal checkpointing, and the enterprise-adoption safety envelope (fail-closed MutationGate, hash-chained ledger, human approval, breaker kill-switch) are genuinely implemented and in places exceed the source. But the source's headline pattern — a transaction coordinator that executes a COMPLEX multi-step plan as an atomic, cleanly-rollback-able unit — is absent: each execute runs a single argv, and TG has no plan-level all-or-nothing/compensation coordinator (exactly the predecessor's still-open G7). The 'agent undo stack' exists only as a RECORDED compensating inverse in the ledger (one op, re-restart stand-in inverse, no pre-state capture) that no code ever applies, despite comments implying a reconciler runs it.

## Top gaps

- Multi-step atomic transaction coordinator: execute a plan as one all-or-nothing unit and compensate/roll back on partial failure (closes the predecessor's open G7)
- Applied undo executor: consume the recorded execution_log inverse to actually revert an action (console revert + reconciler auto-revert on DEVIATION), not just record it
- Pre-mutation state capture (checkpoint-before-change) so a deviation verdict can trigger a genuine state-restoring rollback
- Broaden the reversible-op library beyond the single restart-service canary with real state-restoring inverses instead of re-restart stand-ins
