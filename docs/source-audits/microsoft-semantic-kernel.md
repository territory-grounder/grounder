# Microsoft — Semantic Kernel

> **[O] audit-overlay · TG-38 source benchmark.** Category: **Microsoft** · TG adherence: **4.2 / 5** · Practices audited: **5** (implemented 3 · partial 2 · gap 0 · N/A-by-design 0).

## Verdict

TG is exceptionally strong on the three safety-critical SK practices and partial on the two instrumentation/extensibility ones. It nails the tool-count anti-pattern (the agent sees exactly 4 read-only, semantically-consolidated tools — well under SK's "ideally <10"), is a pure native function-calling ReAct loop with zero planner/Stepwise/Handlebars residue (matching SK's "function calling replaced planners" insight verbatim), and implements and exceeds "reads-safe vs writes-need-approval" via a structurally read-only tool set plus a wired interceptor Do() chain (approval + never-auto floor + evidence + verifiability gates). It falls short of SK's OpenTelemetry mandate: observability is real (Prometheus /metrics, self-telemetry, freshness-stamped samples, optional Langfuse/OpenObserve, Temporal durable history) but there is no OTel SDK, no distributed tracing spans across the prompt lifecycle, and no per-session agent metrics (tool-call count, tokens, tool errors, latency). The SK 3-level filter pipeline exists in spirit — function/prompt/auto-invocation-stage filters are all present and wired — but as bespoke hardcoded gates, not a composable, registerable pipeline.
