# Writing Effective Tools for Agents

> **[O] audit-overlay · TG-38 source benchmark.** Category: **Anthropic** · TG adherence: **3.75 / 5** · Practices audited: **8** (implemented 4 · partial 4 · gap 0 · N/A-by-design 0).

## Verdict

TG treats its read-only agent tools as a real ACI, and on the highest-leverage items it meets or exceeds the source. Semantic observation IDs (lnms-dev-<host>, estate-ctx-<host>) and actionable, fail-soft error strings are not just present but load-bearing — the citation gate requires the model to copy those semantic ids verbatim, operationalizing the source's anti-hallucination finding as a correctness check. Consolidation is honored (get-estate-context folds three cascade questions into one call; every LibreNMS tool returns host-scoped data, never a full list), and the ACI was demonstrably iterated via evals. Where TG falls short is the parameter surface and the eval rigor: tools are passed to the model as a bare name list plus skill prose (no structured tool-calling schema in adapters/model/model.go), args and proposal op_class are free-text rather than enums (constrained only downstream), there is no agent-selectable response_format, and the 'sealed holdout the system may never tune to' remains Planned rather than built.

## Top gaps

- Surface typed per-tool parameter schemas + examples to the model (structured tool-calling), not just a comma-list of tool names + skill prose
- Constrain tool args and proposal op_class with enums/typed schema at emission (poka-yoke), reusing the actuator's reversible allowlist as the op enum
- Stand up the sealed held-out agentic tool-eval set the flywheel may never tune to, with the regression-vs-holdout overfitting alarm
